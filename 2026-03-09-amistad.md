We don't really have holidays, today, but in the United States, we celebrate the 185<sup>th</sup> anniversary of the ruling in [*United States v Amistad*](https://en.wikipedia.org/wiki/United_States_v._The_Amistad), ruling that the escaped slaves emphatically did not qualify as anybody's property, nor did they commit any crimes by turning on their captors.  Historians often see it as one of the most important and influential "freedom suits," alongside [*Dred Scott v Sandford*](https://en.wikipedia.org/wiki/Dred_Scott_v._Sandford).

## Entropy Arbitrage

{% codeberg jcolag/entropy-arbitrage-code %}

I'll go into more detail in a full [tech tips](/blog/tag/tech-tips) post soon, but the blog now should have the start of a more maintainable color palette...again.

You might *vaguely* remember that I tried this [once before]({% post_url 2023-04-03-yuknoom %}) several years back, converting all the color definitions to the [HSL](https://en.wikipedia.org/wiki/HSL_and_HSV) color space, which I have tried to use and recommend where I can, because they make more intuitive sense.  Once you learn that a hue of zero degrees means red, a hundred twenty degrees means green, and two hundred forty degrees means blue, you can reason about what an arbitrary color probably looks like, based on its saturation (boldness) and lightness.  For example, a hue of 300°, 30% saturation, and 80% lightness probably looks lavender, right?  Between blue and red, faded, but light, so washed out a bit.

However, HSL has its flaws, which I'll skip over for now, except to say that it still requires a lot of visual inspection at every step.  Therefore, enter [okLCH](https://en.wikipedia.org/wiki/Oklab_color_space), which has a similar cylindrical shape, but aims for perceptual uniformity, so that every hue should "feel" the same for the same lightness and chroma (boldness, analogous to saturation).  It has its own flaws, such as "gaps" in the cylinder where it can't guarantee that perceptual uniformity, but it works widely in CSS with that caveat.

Maybe more importantly, but definitely fitting well with okLCH colors, CSS also now widely supports [relative colors](https://developer.mozilla.org/en-US/docs/Web/CSS/Guides/Colors/Using_relative_colors).  You can now say "the background color, but ten percent lighter" or similar variations.

As I have written about before, [mathematical color schemes]({% post_url 2020-03-11-colors %}) often work well, Ethan Schoonover's [Solarized](https://ethanschoonover.com/solarized/) probably providing the most famous example.  For example, the blog colors start from the blue text (blue background, for those of you reading in dark mode), then take the complement (180° away) plus or minus thirty degrees to get the yellow and red, what the design people call "analogous colors with complement," except that I don't use the "main" orange-y color that complements the blue.  If I wanted to shift the blue more towards the purple or the green, then the yellow and red should follow to maintain the "harmony," as much as you can call it that.

For an example of where this could prove useful, Jekyll allows me to add metadata to posts in the "front-matter," which the layout templates can use to generate the post.  In theory---I don't envision myself actually doing this---I could add a field that sets a custom "root color" for that specific post and only there.  If I didn't set anything, the post would leave the default color in place.  But if a (almost seasonally appropriate, after all) St. Patrick's Day post set the root color to green, then the `from` operator would automatically fill in a pink and violet that probably match fairly well.  A hypothetical Valentine's Day post (quasi-seasonal last month) could set a red/pink as the base color, and have a matching green and blue fall out without needing to specify them.  All would use the same effective color scheme, but "rotated" to fit a hue that has some relevance to the day.

Now, my color scheme doesn't yet all swing from some root color like that, but it has narrowed to four---the blue, red, yellow, and gray---with everything else altering the lightness and chroma of one of those.  And maybe more importantly, testing exposed a variety of choices that didn't work, such as changing the link colors or some muddy contrasts.  You shouldn't spot any glaring changes to the major color choices, but those ugly edge-cases should look better, such as links consistently showing as blue, visited links always purple, and "active" links red, consistent with web browser defaults.

This also lays the groundwork to eventually start removing the [SASS](https://sass-lang.com/) variables for at least the colors, since SASS, at least the version used by Jekyll, increasingly feels like a hindrance not keeping up to date than a productivity tool.
