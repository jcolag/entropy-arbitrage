Last week, I mentioned [noting my solidarity with Wikimedia's workers]({% post_url 2026-06-01-children %}) during an employee purge to stave off unionizing, proving the need for a union even at the Wikimedia Foundation.  In particular,last time, I signed the [collective action petition](https://en.wikipedia.org/wiki/Wikipedia:Wiki_Workers_United_solidarity) opposing the firings and threatening collective action, quickly catching up to the site's largest petition standing against supporting protecting editor anonymity from government demands.

This time, checking the [Wiki Workers United](https://wikiworkersunited.org/take-action-community/) action recommendations, they point to the separate [solidarity statement](https://meta.wikimedia.org/wiki/Wiki_Workers_United#c-Jcolag-20260604130300-Individuals), which I have also signed (with the same statement).  And at that page's recommendation, [my user page](https://meta.wikimedia.org/wiki/User:Jcolag) includes support for Wiki Workers United.

Should this dispute come to collective action, I'll probably find one of the independent Wikipedia mirrors to link to for the duration, maybe doing so retroactively.  After all, I *could* quickly write a Jekyll plugin that turns `wiki Wikipedia:Wiki_Workers_United_solidarity` (for example) into the petition URL from two paragraphs back, though rewriting over six thousand links feels possibly error-prone, and then replace the site if the need arises.  I don't *intend* to replace Wikipedia as a source here, but if the workers want to start pushing people away from Wikipedia temporarily to make a point, I'll happily find a way to support them, and would rather not use the blog to encourage people to cross the picket line.

Seriously, I want Wikipedia and the Wikimedia Foundation to thrive, as I have said many times before.  But it can *only* thrive if it treats its employees and community well, something that the last weeks have proven can only consistently happen if something (yes, such as a union with the community backing it) forces their hand.

### Simplifying Newsletter Updates

Less visible for a few weeks, and ideally never *actually* visible to casual readers, I have implemented the two [text filters]({% post_url 2026-06-03-jekyll-text-filters %}) that I wrote about in Wednesday's post.  One filter takes a date and a day of the week and returns the first instance of that day of the week starting at the last day of the current month.  It'll save me the trouble of remembering to look up---again I mean, since I did it for this example sentence---that the July 2026 newsletter should go out on August fourth.

Oh, somebody might need that calculation, someday, so I should probably talk about a simplified version of that, after some upcoming fixes.

```ruby
# date:  An input date
# dow:  An integer from 0 to 6
day1 = Date.new date.year, date.month + 1, 1
wday = day1.wday

wday == (dow + 1) % 7 ? day1 - 1 : (day1 + dow + 7 - wday) % 7
```

We find the first day of the next month, taking the year from the date passed in, the month-plus-one, and day-one.  Then we check if the day of the week---0 for Sunday through 6 for Saturday---for the first of the next month matches the day after the requested day, Wednesday in our case.  If it does, then return the day before the first of next month (the last day of the current month), because the month ends on a Tuesday.  If the month doesn't start right after our target, take the first of the month, plus the requested day of the week, minus the day of the week of the first, normalized to the zero-to-six range.

In other words, for that latter calculation if we ask for the first Tuesday, you get the following possible values.

|Day of Week|N|Tuesday|
|-----------|-|-------|
|Sunday     |0|3|
|Monday     |1|2|
|Tuesday    |2|1|
|Wednesday  |3|0|
|Thursday   |4|6|
|Friday     |5|5|
|Saturday   |6|4|

It "breaks" for Wednesday, but we already solved that in the former calculation to get the last day of the month.  Otherwise, for example, if the month starts on a Saturday, then the first Tuesday comes on the fourth.  If we had asked for the first Sunday instead, instead of adding two to the first of the month, we'd add zero, so each value in that last column on the table gets shifted up by two cells.

The other filter, not really for general use beyond working with the first filter, turns an input number into the ordinal term, first, second, third, and so forth.  Because I don't (yet?) need it for anything other than the first specified-day-of-week of a month, it currently only covers the first through the sixth, then the twenty-eighth through the thirty-first.  More general than that, it'd warrant an algorithm to check each place and look up the relevant values, then figure out whether to use the cardinal or ordinal form for that place ("fortieth" versus "forty-second"), which I could do, but with only ten possible values, that seems silly.

The filters will see action at the end of the month.  And since the text (ideally) almost always comes out identically from month to month other than the details specific to a given month, I could probably extract it as its own included file where I can supply the unique parts and forget about the rest.

### Colors

Visible to a subset of readers, all pages now include the pre-CSS color scheme directive in the page header.

```HTML
<meta name="color-scheme" content="light dark" />
```

Let me explain that subset, since you all know that I can't single anybody out:  You know that horrifying moment when you head over to a page and the browser briefly flashes white or black before loading the page, and always gets it wrong for the color scheme on the page that you left and have coming?  Actually, that sounds too confusing.  For a concrete example, imagine that, reading in the middle of the night, you have a page with dark colors up so that your reading doesn't light up the room.  You finish reading and click over to another dark site.  What happens?  You might often get a bright white flash while the browser decides what to do about the new website.

The `color-scheme` directive above tells the browser that, once loaded, the page style will reflect the user's light/dark preference, so it (the browser) should plan for that.  You can also set it to only "light" or "dark," if your page works that way.

I hope that change stresses people out a bit less.

In addition, also something that only certain readers will notice, the pages should also appropriately set the [theme color](https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Elements/meta/name/theme-color), which changes some controls and borders (the "chrome") on some browsers, such as the system-level controls (the application list, "desktop," and back buttons) on Android browsers.  It looks like this, if you want to play with it.

```HTML
<meta name="theme-color" content="#d9cea2" media="(prefers-color-scheme: light)" />
<meta name="theme-color" content="#143048" media="(prefers-color-scheme: dark)" />
```

Possibly the least-important thing that I could take care of, but it gives sites slightly more of an app-like appearance.

### Icons

Probably less exciting in general, but the site "favorites" icon should look cleaner on most platforms.  It looks like the first version of my personal website launched in (probably) 2013 with nothing more than the styles and a note that I'd populate it "soon," with some actual text sometime in 2014.  Since then, I have used a 32-by-32 pixel version of my little JC icon.

Over the weekend, that should have changed.  Most browsers can now use an SVG file, except on mobile, where they expect (variously) 180-, 192-, and 512-pixel versions for the (almost certainly rare) cases of people adding the site to their home screen.  If you look in the page headers, you'll see something like this.

```HTML
<link rel="shortcut icon" href="/blog/assets/favicon.svg" type="image/svg+xml">
<link rel="shortcut icon" href="/blog/assets/jc.ico" sizes="32x32">
<link rel="apple-touch-icon" href="/blog/assets/favicon-180.png.avif" />
<link rel="manifest" href="/blog/assets/site.webmanifest" />
```

The original icon still there, but joined by the SVG, the "medium-sized" version for Apple, and then the manifest for (I guess) Android.

```JSON
{
  "icons": [
    { "src": "/blog/assets/favicon-192.png.avif", "sizes": "192x192", "type": "image/png", "purpose": "any" },
    { "src": "/blog/assets/favicon-512.png.avif", "sizes": "512x512", "type": "image/png", "purpose": "any" },
    { "src": "/blog/assets/favicon-512.png.avif", "sizes": "512x512", "type": "image/png", "purpose": "maskable" }
  ]
}
```

If the AVIF images don't work, somebody tell me.  I took that particular leap of faith largely because I didn't feel like messing with my existing asset-management script to *not* convert three images.
