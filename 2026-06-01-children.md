## Wikipedia

A long time ago, I signed up as a Wikipedia editor, and ran down some citations and fixed links.  As I recall, I chose to do so *primarily* because somebody once did enough research to create an entry on me, and it seemed worth having some presence so that I could credibly vote for deletion, if it ever came up[^EMYz1N], but it also seemed worth the energy of supporting the project.  After mostly forgetting about it due at least partly to a lack of time, I have recently started thinking of returning to the fold and doing more serious editing.

[^EMYz1N]:  The existence of the page didn't offend me, and it certainly didn't reveal anything that a web search at the time wouldn't turn up---that I taught and created some silly programming languages, one of which became mildly popular---so I had no objection to it staying or going, but I also can't imagine myself as notable, since people don't write about me, as far as I know, and felt an obligation to make myself available if it came up.  As it turns out, that didn't matter, because the few official links substantiating the article went stale, so it got purged along with many other "we can't find anything published about this person and the citations don't go anywhere" articles.

Then, last week, news broke that Wikimedia's new CEO---fresh off a finance career---has begun firing developers and community managers, because they wanted to unionize.  That has put off my interest in returning to editing, **except** for [lending my voice](https://en.wikipedia.org/wiki/Wikipedia:Wiki_Workers_United_solidarity#c-Jcolag-20260526221900-Tamzin-20260521135800) to the growing solidarity movement.

If you have a Wikipedia account, you can do the same; sign your comment with `~~~~`, which inserts the link to your user page and time stamp.  If you don't have a Wikipedia account, you can *create* one and do the same, which in addition to letting you edit pages also gives you some options on displaying pages.

## Scrawls

{% codeberg jcolag/scrawls %}

Yes, I should have gotten myself waist-deep in Goblins by now, but a flash of quasi-inspiration came to me.  As I use my home server(s), the setup has one major flaw, if I think of it as partly a backup system:  The blog doesn't have a convenient workspace for me, should this laptop have problems.

Granted, the blog *can* have a workspace thrown together with a few minutes of labor, pulling everything that I need from the different git repositories, and (depending on conditions) a couple of hours of waiting for image conversions.  If necessary, a terminal shelling into the server can edit the files with `vi`, and it'll work.

However, I don't work that way, making that unappealing.  On my desktop, I usually have the next few weeks' worth of posts, about half the draft posts, and the main CSS file, all open in `gEdit` at any given time with whatever else I might want to work on.  As I have ideas for different posts or "scaffolding" to put up to make the writing quicker, I bounce from file to file.  It works for me, and jumping in and out of the editor or dealing with a terminal editor's multi-document interface won't cut it.

Looking for a web-based editor, I couldn't find anything useful, with most preferring (for example) that the browser use its tabs to handle multiple documents, which becomes clutter among everything else that I have open.  Several will also let me edit filed on my local computer, as if I somehow couldn't find any way of changing a text file on my operating system.  And that, to my chagrin, means writing my own.

### Nowhere Fast

Oh well, it at least gives me an excuse to play with different technology, I suppose, such as building a service in Go.  You might know that I have largely objected to Go on principle.  Google has the kind of control over it that Microsoft has (had?) over C𝄱, and I have already worked in that sort of space.  They introduced the language as having a *fast compiler*, as if I care how much effort my *CPU* puts into getting the software ready to ship[^xqIzaC].  That introductory video went on to make a big deal about the ability to read code aloud, from left-to-right, as if actual humans do that, reading code to each other like poetry, but getting stuck because sometimes the symbols don't come in the right order.  They not only gave the language such a worthless name that people make it even worse by calling it "Golang"---presumably to distinguish it from *gomo*, when "go" refers to moving, which I similarly made up---but they picked a name openly conflicting with a fairly known language, and largely dismissed the conflict on the basis (in essence) that these sorts of things happen all the time and the more powerful organization wins, so tough luck.

[^xqIzaC]:  By the way, why has nobody written a compiler that compiles individual lines as mini-object files to link together as a single long program later, allowing a compiler to recompile *as you type*?  I prototyped something like that in college decades ago, so let's not pretend that the technology doesn't exist...

Design-wise, because I *have* given it a look, I think of it as a language thirty-to-forty years too late, in many ways feeling like they went through programming texts from the 1980s and 1990s to find criticisms of C and built that [faster horse](https://en.wikiquote.org/wiki/Henry_Ford#:~:text=faster%20horses) instead of looking at the future.  While competitors such as Rust look at where software fails, and how to keep developers from getting into those situations, Go boldly discards semicolons and says that the brackets should go before the type so that you read it as "array of [whatever things]."

It has ideas that I like, mind you.  It never should've taken until the twenty-first century, for a programming language to impose a style, saving every dopey argument about spaces (and how many) or tabs, capitalization, or where the braces sit.  Including a static analyzer and a unit testing framework seems sensible, too.

And when I look at the various systems that I run on my home servers, the better-written Go applications have the least work involved.  Tell Nginx to map a folder name to the port number, and done, which I find appealing.

### Upshot

Most likely, I won't go all-in on this project, but rather make it a "spare time in my spare time" kind of deal.  I don't have a pressing need for it, but the need will eventually arise, and I'd like to have it ready.  And unlike many of my other projects, maybe someone else will find this useful as it gets moving.

At this point, it can open a single file and show the contents in a `textarea` element.

## Entropy Arbitrage

{% codeberg jcolag/entropy-arbitrage-code %}

I still don't know if I have a real use for this, but on a whim, I finally took the final steps that'll allow me to occasionally have a post that "rotates" its color scheme.  Specifically, the set of hues that the styles work with all derive from the red color generally used in headings.  And if I add a `root_red` value to the post's front-matter/metadata, the browser will run through the calculations again and update the page.

Again, I can imagine using this for thematic posts.  For a typical example, maybe the traditional winter holiday story drawn from the public domain might work better with a green or blue background instead of the yellow-brown (blue in dark mode) on other pages.  It also works as an illustration, though, for people reading this on the website and not RSS.

<script type="text/javascript">
function changeRed() {
  const slide = document.getElementById('color-change');

  document.documentElement.style.setProperty('--red', slide.value);
  slide.title = slide.value.toString();
}
</script>
<input
  id="color-change"
  list="markers"
  max="359"
  min="0"
  oninput="changeRed()"
  style="left: 50%; position: relative; transform: translateX(-50%); width: 70%;"
  title="36"
  type="range"
  value="36"
/>
<datalist id="markers">
  <option value="6"></option>
  <option value="36"></option>
  <option value="66"></option>
  <option value="96"></option>
  <option value="126"></option>
  <option value="156"></option>
  <option value="186"></option>
  <option value="216"></option>
  <option value="246"></option>
  <option value="276"></option>
  <option value="306"></option>
  <option value="336"></option>
</datalist>

With JavaScript enabled, you can use the slider to change the "red" color to any hue between zero and three hundred fifty-nine and watch the entire color scheme change, which I suspect demonstrates the idea better than me setting the red in this post to a hue of 126 or whatever, even though the actual use makes no sense.  The tick-marks prioritize the values of 36 (the standard red) plus or minus multiples of thirty degrees.

To recap how this all works, I have (something like) the following in my CSS (technically SASS) to define the colors by hue.

```CSS
:root {
  --red: 36;
  --yellow: calc(var(--red) + 60);
  --blue: calc(var(--red) + 210);
}
```

If I set `root_red` on a post, that creates a new `:root` style, which (defined on page) overrides the existing red.

Those hues then go into OKLCH colors, which---in light/dark pairs---get semi-semantic names for use through all the styles.  Because CSS works kind of like a spreadsheet application, where every change triggers a round of recalculation, changing `var(--red)` changes `var(--yellow)` and `var(--blue)`, which changes the OKLCH colors, changing the display.

On this page, the slider calls an event handler that---skipping the boring parts that you can see by looking at the page source---does this.

```JavaScript
document.documentElement.style.setProperty('--red', slide.value)
```

And that lets you rotate through all the hues.  And going back to my post about [updates on CSS colors]({% post_url 2026-03-18-colors-revisited %}), by using hues (color harmonies) with a clear mathematical relationship, it always looks similar, no matter the actual colors.

I see two flaws.  First, because I change the colors in dark mode, changing the background to blue in light mode will change that to a different color in dark mode.  Second, I still believe that, unless you have a *fantastic* excuse to do something else, hyperlinks should always look blue, red (when clicked), or purple (for places that you've visited) so that people (with color vision, at least) have that cue; the current system ties all the colors together, so changing the red will change the link colors.
